# ADR-0030: Role-Based Access Control with Permission-Level Policies

## Status

Accepted

## Context

Up to Phase 12 the only authorization primitive in the codebase is "authenticated or not". Every protected endpoint calls `.RequireAuthorization()` with no policy, or the `Authenticated` baseline policy. That is sufficient to prove the auth pipeline works but is not sufficient for a general-purpose template: nearly every real application needs at least an "admin vs. user" distinction, and most grow past that to finer-grained capability checks.

We want to add authorization without importing an entire identity/authorization framework and without building something that has to be torn out later. The constraints that shaped this ADR:

1. **The Users module is lean by deliberate choice (ADR-0007).** We are not adding ASP.NET Identity. Any RBAC model has to fit the lightweight custom aggregate shape the Users module already uses.
2. **Modules are independent.** A module's endpoints should be able to declare their own authorization requirements without the Users module knowing what they are.
3. **Contracts are the only cross-module surface.** If endpoints in the Catalog module authorize on a permission named `catalog.products.write`, that string has to live somewhere the Catalog module owns.
4. **Tokens should stay small.** Per-request HTTP response size is bounded in practice by the `Authorization` header size acceptable to load balancers and proxies (typically 8 KB total). Baking a large permission list into every JWT is not free — at scale it shows up in egress bandwidth and in corner-case header-size failures.
5. **The model should be simple enough that architectural tests can enforce its invariants.** Complex authorization graphs (hierarchical roles, dynamic rule evaluation, resource-scoped grants) are where authorization systems become impossible to reason about.

## Decision

Introduce a role-based access control model with the following shape.

### One role per user

Each `User` has exactly one `Role`. Roles cannot be granted or revoked — only *changed*. The domain operation is `User.ChangeRole(newRole, changedBy)`, not `AssignRole` / `RevokeRole`.

This removes an entire class of questions from the system:

- No junction table, no `UserRoles` collection.
- No "what roles does this user have today" query; it is a column on `users`.
- No partial-grant race conditions.
- The claim surface on the JWT is trivially one `role` claim.
- The audit event is always `RoleChanged` (old → new), never `RoleGranted` / `RoleRevoked`.

We accept the constraint this puts on modelling — users cannot hold multiple orthogonal roles (e.g., "catalog editor" *and* "audit reader"). For a template this is the right trade. Applications that genuinely need multi-role users can introduce a join table later; it is a well-understood migration. Shipping single-role simplifies the default.

### Roles → permissions

Endpoints authorize on **permissions**, not roles. A permission is a dotted string such as `catalog.products.write`. Roles are static compositions of permissions, defined in code.

Two roles ship by default:

- `admin` — holds every declared permission across all modules.
- `user` — holds no permissions. Endpoints that only require authentication continue to use the existing `Authenticated` policy.

Adding a role (say, `editor`) is a code change — a new entry in the role→permission map. No schema change, no data migration. We are not optimising for runtime role administration because runtime role administration is where most RBAC systems accumulate accidental complexity.

### Permissions are declared by modules in `.Contracts`

Each module declares its permission constants in its `.Contracts` project:

```csharp
// Modulith.Modules.Catalog.Contracts/Authorization/CatalogPermissions.cs
public static class CatalogPermissions
{
    public const string ProductsRead  = "catalog.products.read";
    public const string ProductsWrite = "catalog.products.write";

    public static IReadOnlyCollection<string> All { get; } =
        [ProductsRead, ProductsWrite];
}
```

This places the authorization surface on the same side of the module boundary as the module's other public contracts (events, request/response DTOs). Endpoints inside the module and subscribers outside it reference the same constants.

Format: `{module}.{resource}.{action}`, lowercase, ASCII. Enforced by architectural test.

### Permissions resolved from a static in-memory map; JWT carries only the role

The JWT access token carries exactly one authorization-relevant claim: `role`. Permissions are *not* baked into the token.

On each authenticated request, an `IClaimsTransformation` reads the `role` claim, looks up the permission set from a static in-memory `role → permission[]` map, and adds `permission` claims to the cloned principal for the lifetime of the request. Authorization handlers check for the relevant `permission` claim.

Why not bake permissions into the JWT:

- **Token size.** An `admin` role for a system with dozens of modules can carry hundreds of permissions. Baked into the token, that inflates every request.
- **Staleness window.** Baked permissions are stale until the access token expires. If a release adds a new permission or reshapes the map, existing tokens do not see the change until refresh.

Why not resolve permissions from the database per request:

- **No new data.** The map is defined in code, not the database. There is nothing to query.
- **Cost.** Even cached, it is work for no benefit over the in-memory lookup.

The trade-off this locks in: changing which permissions a role holds is a **code change**, not a data change. We consider this a feature, not a limitation. Permission composition is part of the system's authorization contract, and it belongs in the source tree next to the endpoints it governs. An admin UI that edits role→permission mappings is the wrong abstraction for a system with a finite set of roles defined by the application.

### Role change forces re-login

When a user's role changes, the server **revokes all of that user's refresh tokens** (per ADR-0029). Their current access token continues to work until it expires (at most `AccessTokenLifetimeMinutes`, default 15). On the next refresh, they are forced back through login, and their new token carries the new `role` claim.

This avoids every complication around "stale role in token": there is no revocation list to check, no per-request DB round-trip, no claim-refresh middleware. The access-token TTL bounds the stale-role window. Operators who need that window shorter should shorten the access-token lifetime — the control is already there.

### `ICurrentUser` exposes role and permissions

The shared kernel abstraction gains:

```csharp
string? Role { get; }
IReadOnlyCollection<string> Permissions { get; }
bool HasPermission(string permission);
```

`Role` reads `ClaimTypes.Role`. `Permissions` reads all `permission` claims. `HasPermission` is a convenience. Handlers that need to *do something conditional* on the caller's permissions use these. Handlers that need to *gate* an action rely on the endpoint's policy, not an in-handler check.

### Admin bootstrap

Two bootstrap paths, neither requires the other:

1. **Development seeder.** `UsersDevSeeder` ensures an admin exists in dev. Email and display name from `Modules:Users:Dev:AdminEmail` / `Modules:Users:Dev:AdminDisplayName`. Idempotent.
2. **Non-dev `AdminBootstrapper : IHostedService`.** Registered only when `Modules:Users:AdminBootstrap:Enabled` is `true`. On startup:
   - If any admin exists → no-op.
   - Else if a user with the configured email exists → promote.
   - Else → warn and continue (will not create a user out of thin air — no password, no email confirmation).
   - If the option is enabled but `Email` is missing → fail fast.

This gives operators a controllable, auditable way to promote the first admin without baking credentials into the image.

### Endpoints that do not need permissions

Endpoints scoped to the caller (`/v1/users/me`, `/v1/users/me/password`, `/v1/users/me/email/*`, personal-data export, account deletion) continue to use the `Authenticated` policy. They authorize on *who you are*, not *what you may do*. Introducing a permission for them would be noise.

### Delivering the permission set to the frontend

Frontends need the caller's permissions to gate UI (show/hide buttons, routes, menu items). The JWT intentionally carries only `role`, so the server exposes the resolved set through `GET /v1/users/me`:

```json
{
  "id": "…",
  "email": "…",
  "displayName": "…",
  "role": "admin",
  "permissions": ["audit.trail.read", "catalog.products.read", "catalog.products.write", "…"],
  "permissionsVersion": "a1b2c3d4e5f60718"
}
```

Rationale for folding this into `me` rather than a dedicated endpoint:

- `me` is the canonical "hydrate this session" call. Frontends already invoke it after login and refresh. Piggy-backing avoids an extra round-trip.
- Permissions change for a user only when their role changes (per ADR-0030), and a role change revokes refresh tokens (per ADR-0029). The frontend's cached `me` response is therefore always as fresh as it can be — re-login is the moment of truth.
- A separate `GET /v1/auth/permissions` endpoint would be cleaner in isolation but adds a second source of truth for "who am I" and tempts inconsistency.

Contract guarantees:

- `permissions` is sorted lexicographically and never `null`.
- `permissionsVersion` is a short stable hash of the sorted permission list for the caller's role, cached in-process. It changes when the role→permission map changes (i.e., on deployment). Frontends can use it to short-circuit equality checks and invalidate cached UI state.
- The response is not cacheable by shared caches: `Cache-Control: private, no-store`.

Exposing a role-indexed permission catalog (`GET /v1/auth/roles/{role}/permissions`) or a full permission catalog (`GET /v1/auth/permissions/catalog`) is **not** shipped by default. Both are reasonable additions when an admin UI needs them; neither is required for UI gating.

## Consequences

### Positive

- A complete RBAC story ships with the template, with no new infrastructure dependencies.
- The authorization surface is declarative: grep `*Permissions` to see everything a module can gate on.
- Tokens stay small and role changes take effect within one access-token lifetime.
- Architectural tests can enforce permission naming, module ownership, and endpoint conventions mechanically.
- Audit gets a single clean event (`UserRoleChanged`) for all role transitions.

### Negative

- Users cannot hold multiple orthogonal roles without a schema change. Teams needing multi-role users have a straightforward migration path but cannot do it by configuration.
- Runtime role-permission editing is not supported. Changing what `Admin` can do is a code change and a deployment.
- The `permission` claims are added per-request via `IClaimsTransformation` — a trivial cost but not free.

### Neutral

- Role count is expected to stay small (2–5). Tools and documentation assume this. A system with 40 roles will work but will feel under-featured; that is by design.

## Alternatives considered

### ASP.NET Identity (+ `RoleManager`)

Rejected for the same reason ADR-0007 rejected ASP.NET Identity for the core user aggregate: too much surface area for a template, forces opinions we do not want to make default (EF schemas, stores, managers, UI assumptions).

### Permissions in the JWT

Rejected for token size and staleness. Discussed above.

### Permissions in the database (per-role table)

Rejected for a single-role-per-user model where the set of roles is small and defined by code. Adding a table that only the seeders and a hypothetical admin UI would ever write is overhead without payoff.

### Multi-role users with RBAC join table

Rejected as a default because it moves the authorization model away from "a property on the user" toward "a collection requiring joins and cache invalidation". Single-role is a ceiling; the ceiling is appropriate for a template.

### Claims-based policies without roles

Rejected because it removes the one concept every stakeholder already understands ("Alice is an admin"). Roles are the user-facing unit even if permissions are the runtime unit.
